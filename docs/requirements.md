## 1. 背景と目的

Honmono協会のAI活用勉強会（登録30名超、実働約10名）で、
メンバーが共同開発できる「便利ツールの土台」を作る。

- メンバーそれぞれが自分の環境・自分のAIツールでガジェット（小機能）を開発できる
- 作られたガジェットはプラットフォーム上で公開され、各ユーザーが取捨選択して自分の画面に実装できる
- 将来的に完成度が上がれば、Honmono協会のプロダクトとして中小企業向けに販売する

## 2. スコープの分離

| レイヤ | 内容 | 触る人 |
|---|---|---|
| プラットフォーム | 認証、ロール管理、ガジェットカタログ、ガジェット実行基盤、共通ストレージ | platform開発者（少人数） |
| ガジェットSDK | ガジェットとプラットフォームの通信契約 | platform開発者が提供、全developerが利用 |
| ガジェット | 個々の便利機能（タスク管理、スケジューラ、メモ、etc.） | developerロールの全メンバー |

## 3. ユーザーロール（4種）

| ロール | 定義 | 主な権限 |
|---|---|---|
| admin | プラットフォーム運営者 | 全権。ガジェット審査・公開、ロール付与、緊急停止 |
| developer | ガジェット開発メンバー | ガジェット登録・更新・公開申請、自作ガジェットの利用統計閲覧 |
| user | 勉強会メンバー（一般） | ガジェットのインストール・利用・個人設定、自分のデータ管理 |
| guest | 招待前・体験者 | 公開カタログとデモの閲覧のみ。データ保存不可 |

- ロールは上位が下位を包含する（admin ⊃ developer ⊃ user ⊃ guest）
- ロール付与はadminのみが行う。初期adminは向井

## 4. 機能要件

### 4.1 プラットフォーム（v1で必須）

- FR-01: メール（マジックリンク）およびGoogleアカウントでのログイン
- FR-02: ロールに応じた画面・API・データアクセス制御（RLSで強制）
- FR-03: ガジェットカタログ（一覧、詳細、必要権限の表示）
- FR-04: ユーザーごとのガジェットのインストール／アンインストール
- FR-05: ダッシュボード（インストール済みガジェットの配置・表示）
- FR-06: ガジェットごとの設定画面と権限承認フロー（初回起動時に必要権限を提示し、ユーザーが承認して初めて有効化）
- FR-07: ガジェット用のユーザー別ストレージ（key-value、ガジェット間で隔離）
- FR-08: developerによるガジェットの登録・バージョン更新・公開。
  **勉強会フェーズは事前審査を行わず、事後統治とする**:
  ①CIによる機械チェック（マニフェスト検証・宣言外API呼び出しの静的検査・ビルド）
  ②インストール時のユーザーによる権限承認（FR-06）
  ③adminによる緊急停止（FR-09）。
  B2Bフェーズで「admin事前審査・署名済みガジェットのみ実行可」の署名制に移行する（§8）
- FR-09: adminによるガジェットの緊急停止（全ユーザーで即時無効化）

### 4.2 ガジェット（SDK経由で提供する能力）

- 自ガジェット専用のユーザー別ストレージ read/write
- 表示サイズの要求（resize）
- ユーザーへの通知表示の依頼
- ユーザーのプロフィール最小情報（表示名のみ）の取得 ※承認制
- AI による文章生成（ユーザーがプラットフォームに登録した API キーを消費。
  キーはガジェットに渡さない） ※承認制。詳細は ADR-008
- 外部サービス連携（§5のポリシーに従う）

### 4.3 v1でやらないこと（Non-goals）

- ガジェット間のデータ連携（v2候補）
- ガジェットの有料販売・課金（B2Bフェーズで設計）
- ネイティブアプリ（PWAで代替。ストア配布はしない）
- リアルタイム共同編集
- restricted scope を要する外部連携（Google Driveフルアクセス、Gmail等）

## 5. 外部サービス連携ポリシー（v1）

ガジェットごとに、権限の承認・外部サービスのアカウント作成・支払い手段の登録を
**ユーザー自身が行う**という原則を守る。具体的には:

1. **プラットフォームが仲介するのは、Googleの非restricted・非sensitiveスコープのみ**
   （例: `drive.file` = ユーザーが明示的に選択したファイルへのアクセス）。
   restricted scope（Driveフルアクセス、Gmail等）はプラットフォームとして扱わない。
   理由: CASA年次セキュリティ評価（有償・毎年）の負担と、未検証時の100ユーザー生涯上限。
2. **それ以上の連携が必要なガジェットは BYOK（Bring Your Own Key）方式**。
   ユーザーが自分で対象サービスのAPIキー等を取得し、ガジェット設定画面から入力する。
   キーはユーザー別ストレージに暗号化保存し、該当ガジェットのみが参照できる。
3. ガジェットは必要な外部連携をマニフェストで**事前宣言**する。宣言外の外部通信は審査で却下。

## 6. 非機能要件

- NFR-01 環境非依存: スマホ／タブレット／PC、Windows／macOS／iOS／Android の主要ブラウザで動作。PWAとしてホーム画面追加可能
- NFR-02 コスト: 勉強会フェーズの月額固定費 0円〜1,000円以内（Cloudflare無料枠 + Supabase無料枠を前提に設計）
- NFR-03 セキュリティ: ガジェットは sandboxed iframe に隔離。ユーザートークンをガジェットに渡さない。全データアクセスはRLSで強制
- NFR-04 開発体験: developerは `docs/gadget-spec.md` と雛形だけでガジェット開発を開始できる。platform内部の知識を要求しない
- NFR-05 監査可能性: ガジェットの公開・更新・停止、ロール変更は操作ログを残す

## 7. 開発フェーズ

| フェーズ | 内容 | 完了条件 |
|---|---|---|
| Phase 0 | 本文書群の確定、CLA・ライセンス整備、リポジトリ作成 | メンバーへの周知とCLA同意の取得開始 |
| Phase 1 (MVP) | 認証 + 4ロール + カタログ + サンプルガジェット1個がSDK経由で動く | 勉強会でのデモ |
| Phase 2 | ガジェット登録・審査フロー、ユーザー別ストレージ、権限承認UI | メンバー3名以上が自作ガジェットを公開 |
| Phase 3 | BYOK外部連携、通知、運用ログ | 日常利用の定着 |
| Phase B2B | 署名済みガジェットのみ動作する配布モデル、テナント分離、課金 | 別途要件定義 |

## 8. B2B展開時の前提（今から仕込むこと）

- コントリビューションは全て CLA（貢献者許諾契約）に基づく。CLA未同意のPRはマージしない
- ライセンスは商用権を協会側に留保する（ADR-006参照）。純OSSライセンス（MIT等）にはしない
- B2B版は「admin事前審査・署名済みガジェットのみ実行可」の閉鎖モードを持つ。v1の事後統治（FR-08のCIチェック・ユーザー承認・緊急停止）はその前身であり、B2B移行時に署名制へ切り替える
